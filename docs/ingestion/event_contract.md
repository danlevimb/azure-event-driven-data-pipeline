# Event Contract & Producer

## 1. Overview

The pipeline receives events through Azure Event Hub, generated by a Python-based producer.

This producer simulates order events to test different business scenarios, enabling controlled validation of the pipeline.

---

## 2. Event Structure

Each event follows a consistent JSON contract:

```json
{
  "event_id": "evt-uuid",
  "event_type": "order_created | payment_confirmed | order_cancelled",
  "schema_version": "1.0",
  "source_system": "local-simulator",
  "event_time": "ISO8601 timestamp",
  "payload": {
    "order_id": "ORD-XXXX",
    "customer_id": "CUST-XXXX",
    "order_total": 100.50,
    "currency": "MXN | USD",
    "status": "CREATED | PAID | CANCELLED"
  }
}
```

This structure ensures:

* Traceability (`event_id`)
* Versioning (`schema_version`)
* Separation between metadata and business data (`payload`)

## 3. Producer Behavior

The producer is implemented in Python and sends events directly to Event Hub.

Main responsibilities:

* Generate realistic order events

## 4. Scenario Simulation

The producer includes predefined scenarios to test pipeline behavior under different conditions:

### Valid Scenarios

* `paid_order` → created → paid
* `cancelled_order` → created → cancelled
* `paid_then_cancelled` → full lifecycle
* `open_order` → only created
* `high_value_paid` → large order amount

### Edge Cases

* `zero_amount` → order_total = 0
* `bad_currency` → unsupported currency
* `bad_status` → invalid status value

These scenarios allow validation of:

* Bronze acceptance rules
* Silver business validation
* Gold aggregation behavior

---

## 5. Controlled Testing Capability

The producer supports configurable execution modes:

* Single event generation
* Lifecycle simulation for a single order
* Batch execution with mixed scenarios

This enables:

* Repeatable tests
* Deterministic validation of pipeline logic
* Demonstration-ready datasets

---

## 6. Summary

The producer is not just a data generator—it is a **testing tool**.

It allows the pipeline to be validated against:

* Normal business flows
* Edge cases
* Invalid data scenarios

This ensures the robustness of the ingestion and validation layers.
